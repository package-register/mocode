## 🚨 必须遵守的关键规则

### 架构原则
- **单一职责**: 每个模块/类只做一件事
- **依赖倒置**: 高层模块不依赖低层模块，都依赖抽象
- **接口隔离**: 使用多个专门的接口优于单个通用接口
- **开闭原则**: 对扩展开放，对修改关闭

### 项目结构规范
- **清晰分层**: 按职责分层（如 domain/application/infrastructure）
- **模块独立**: 模块间低耦合，可独立编译和测试
- **命名一致**: 目录和文件命名反映职责和层次
- **文档同步**: 架构文档与代码保持同步更新

### 代码质量标准
- **可读性**: 代码自解释，无需过多注释
- **可测试性**: 无隐藏状态，依赖可注入
- **可维护性**: 修改局部化，影响面可控
- **性能**: 无明显性能陷阱，关键路径有优化

### 技术债务管理
- **可视化**: 债务清单透明，人人可见
- **量化**: 用数据说话，不主观判断
- **优先级**: 按影响和成本排序
- **持续偿还**: 每个迭代预留固定比例

## 📐 架构规范定义

### 推荐分层架构
```
┌─────────────────────────────────────┐
│         Presentation Layer          │  # HTTP/RPC/CLI 接口
│         (controllers, handlers)     │
├─────────────────────────────────────┤
│         Application Layer           │  # 用例编排，不含业务逻辑
│         (use_cases, services)       │
├─────────────────────────────────────┤
│          Domain Layer               │  # 核心业务逻辑，纯 POJO
│         (entities, value_objects)   │
├─────────────────────────────────────┤
│       Infrastructure Layer          │  # 外部依赖实现
│    (repositories, external_apis)    │
└─────────────────────────────────────┘
```

### 模块职责定义
| 层级 | 职责 | 允许依赖 | 禁止依赖 |
|------|------|----------|----------|
| Presentation | 处理外部请求，参数验证，响应格式化 | Application | Domain, Infrastructure |
| Application | 编排用例，事务管理，权限校验 | Domain, Infrastructure | Presentation |
| Domain | 核心业务逻辑，领域规则 | 无 | 任何外层 |
| Infrastructure | 数据库访问，外部 API，文件系统 | Domain | Presentation, Application |

### 命名规范
```
# 目录命名
src/
├── domain/          # 领域层
├── application/     # 应用层
├── infrastructure/  # 基础设施层
└── presentation/    # 展示层

# 文件命名
- 实体：user.entity.ts, order.entity.ts
- 值对象：email.value-object.ts
- 用例：create_user.use_case.ts
- 仓库：user.repository.ts
- 控制器：user.controller.ts

# 类命名
- 实体：User, Order (名词)
- 用例：CreateUser, UpdateOrder (动宾短语)
- 仓库：UserRepository, OrderRepository (名词 +Repository)
- 服务：EmailService, PaymentService (名词 +Service)
```

### 依赖规则
```typescript
// ✅ 允许：外层依赖内层
import { UserRepository } from '../infrastructure';
import { User } from '../../domain';

// ❌ 禁止：内层依赖外层
import { UserController } from '../presentation'; // 编译错误

// ✅ 推荐：依赖抽象
interface UserRepository {
  findById(id: string): Promise<User | null>;
}
```

## 🛡️ 原则与底线

- **不追求完美**: 务实，解决最痛的问题，不追求 100 分
- **不破坏现有功能**: 重构必须保证功能正确
- **不强制推行**: 规范需要团队认同，不是强加
- **不忽视业务**: 架构为业务服务，不为架构而架构
